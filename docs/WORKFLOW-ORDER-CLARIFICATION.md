# Порядок работы по архитектуре — уточнение

Документ фиксирует задуманный порядок и текущую реализацию.

---

## Задуманный порядок (как ты описал)

1. **Включается агент** — генерирует и передаёт данные на портал (клиента/систему).
2. **Клиент** заполняет опросник и загружает нужные документы.
3. **Claude** всё это анализирует и **запрашивает недостающее**.
4. **Клиент** либо дополняет (загружает ещё), либо отмечает, что у него этого нет.
5. **После этого** генерируется отчёт.

---

## Как реализовано сейчас

### 1. Агент → данные на портал

- **Агент** (summit-local-agent) на стороне клиники собирает данные, пакует ZIP (manifest.json, опционально snapshot.json) и шлёт в **Ingest Service** (не напрямую в «портал»).
- **Ingest** принимает пакет, проверяет, пишет **receipt** (ACCEPTED/REJECTED). Данные (manifest_payload, snapshot_data) сохраняются.
- В **SaaS** у тенанта задаётся **client_org_id** (тот же, что агент шлёт в заголовке). Через **proxy API** внутренний портал может показывать список приёмов по клинике.
- При **генерации отчёта** бэкенд подтягивает последний принятый receipt по `client_org_id` и отдаёт его в контекст Claude как **agent_snapshot**. То есть данные агента попадают в отчёт и в анализ Claude.

**Итог:** агент включается, генерирует и передаёт данные; они оказываются «на портале» (в системе) и участвуют в отчёте — по смыслу это совпадает с п.1.

---

### 2. Клиент: опросник + документы

- **Опросник:** клиент на портале заполняет Assessment (ответы на вопросы по контролям), сохраняет, при необходимости отправляет (Submit).
- **Документы:** в Evidence Vault клиент загружает файлы и привязывает их к контролям (теги).

**Итог:** реализовано так, как в п.2.

---

### 3. Claude анализирует и «запрашивает недостающее»

- **Анализ:**
  - **По файлам доказательств:** Claude Analyst (POST `/evidence/{id}/analyze`) — по каждому файлу выдаёт статус (validated / weak / mismatch / unreadable). Вызов **по требованию** (кто-то запускает анализ по файлу).
  - **При генерации отчёта:** Claude получает полный контекст (ответы, результаты движка, гэпы, агрегаты по доказательствам, **данные агента**, предыдущая оценка) и пишет нарратив Executive Summary. То есть «всё это анализирует» на этапе отчёта — да.

- **Запрос недостающего:**
  - Сейчас **запрос документов у клиента** делается **вручную**: консультант (internal) в Evidence Review нажимает **«Request Document»** и создаётся уведомление клиенту (document_request).
  - **Автоматического** сценария «Claude проанализировал и сам отправил запрос на недостающие документы» в коде **нет**. То есть шаг «Claude запрашивает недостающее» в твоём описании сейчас реализован как «консультант запрашивает (при необходимости опираясь на выводы/анализ)», а не как автоматический запрос от Claude.

**Итог:** анализ — да (по файлам и в отчёте); «запрашивает недостающее» — только через ручные запросы консультанта, не автоматически от Claude.

---

### 4. Клиент дополняет или отмечает «нет»

- Клиент видит уведомления (в т.ч. document_request) и может:
  - загрузить ещё документы в Evidence Vault по нужному контролю, или
  - ничего не загружать (фактически «у меня этого нет»).
- Отдельной сущности «отметил, что документа нет» (формальный ответ на запрос) в текущей реализации нет — это выражено отсутствием загрузки или комментарием.

**Итог:** дополнение — да; «отмечает, что нет» — не формализовано отдельным действием в системе.

---

### 5. После этого генерируется отчёт

- Отчёт генерируется **когда консультант решает**: создаёт пакет отчёта → нажимает **Generate** (запускается движок, затем генерация всех артефактов, в т.ч. Executive Summary с Claude) → при необходимости **Publish**.
- Нет автоматической блокировки вида «не генерировать, пока клиент не ответил на все запросы документов». Порядок «сначала сбор данных и ответы на запросы, потом отчёт» держится **процедурой/ролью**: консультант запускает генерацию после того, как считает этап готовым.

**Итог:** да, отчёт делается «после» сбора данных и (при необходимости) запросов, но момент «после этого» задаётся действием консультанта, а не автоматической проверкой.

---

## Сводка: что совпадает, что нет

| Шаг в твоём описании | В реализации |
|----------------------|--------------|
| 1. Агент включается, генерирует и передаёт данные на портал | Да: агент → Ingest → receipt; данные в системе и в контексте отчёта (agent_snapshot). |
| 2. Клиент заполняет опросник и загружает документы | Да: Assessment + Evidence Vault. |
| 3. Claude анализирует | Да: по файлам (Analyst) и при генерации отчёта (полный контекст). |
| 3. Claude **запрашивает** недостающее | Нет: запросы документов только вручную (консультант, «Request Document»). |
| 4. Клиент дополняет или отмечает «нет» | Дополняет — да; явная отметка «нет» — не формализована. |
| 5. После этого генерируется отчёт | Да: генерация по действию консультанта после сбора данных/запросов. |

## Реализовано: шаг «Claude — запросы до отчёта»

- **API:** `POST /api/v1/tenants/{tenant_id}/assessments/{assessment_id}/claude/document-requests` (только internal).
- Действие: запускается движок → собирается полный контекст (ответы, доказательства, **данные агента**) → контекст передаётся **только Claude**. Клиент **не видит** сырой вывод платформы (ControlResult, сырые payload агента и т.д.).
- Claude возвращает список запросов документов (control_code, reason, suggested_document) → по каждому создаётся уведомление клиенту (`document_request`). Клиент видит только эти сформулированные запросы.
- Итоговый отчёт формирует **только Claude** при **Generate** (Executive Summary); клиент получает только опубликованный пакет отчётов.

Порядок проверки: 1) Запустить агента (`.\run-agent-demo.ps1`) → данные в Ingest. 2) Вызвать `POST .../claude/document-requests` → запросы клиенту (если Claude что-то запросил). 3) Клиент дополняет или оставляет как есть. 4) Консультант нажимает Generate → Publish → клиент видит только финальный отчёт.
