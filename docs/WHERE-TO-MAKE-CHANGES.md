# Где вносить изменения: две платформы

Есть **две отдельные платформы**; в одном чате/репозитории делаем только одну.

---

## 1. HIPAA SaaS — главный портал (**этот репозиторий**)

**Репо:** `hipaa-saas` (текущий workspace).

**Назначение:** портал для MSP и клиентов: тенанты, оценки, анкета, доказательства, движок соответствия, отчёты, workforce, приём данных от агента (ingest API / proxy).

**В этом чате делаем изменения только здесь:**
- Генерация Agent Key по тенанту, команда установки (Copy install command), Rotate/Revoke ключа
- Контракт ingest: приём `tenant_id` / `tenant_token` в upload, запись в receipt/package
- Ingest Inbox в портале (таблица receipt’ов, статусы, ошибки, View details)
- Next Best Action (блок «что делать дальше» по тенанту/оценке)
- Runbook (вкладка, 5 шагов) и Checklist по оценке («что нужно для Submit»)
- Любые доработки оценок, доказательств, отчётов, workforce, аудита

**Не в этом репо:** сам агент (сбор, анонимизация, отправка) и код ingest-сервиса приёма пакетов, если он вынесен в отдельный репозиторий.

---

## 2. Агент — сбор, анонимизация, отправка (**другой репозиторий / другой чат**)

**Репо:** отдельный (agent / ingest service).

**Назначение:** локальный агент у клиники/MSP: сбор данных с эндпоинтов, анонимизация (удаление PHI/PII), формирование пакета и отправка на главный портал (через ingest).

**Изменения в том чате/репо:**
- Приём параметров из команды установки (endpoint, ключ, tenant_id / tenant_token)
- Передача в каждом upload заголовка/тела с `tenant_id` или `tenant_token` (по контракту главного портала)
- Логика сбора, санитизации, подписи, повторных отправок
- Любые изменения формата пакета (manifest, dlp_report, checksums), если они инициируются со стороны агента

---

## Сводка

| Что делаем | Где |
|------------|-----|
| Ключи агента, команда установки, Inbox, Next steps, Runbook, Checklist, контракт ingest (tenant в receipt) | **HIPAA SaaS** (этот чат) |
| Установка агента, сбор, анонимизация, отправка с tenant_id/token | **Агент** (другой чат) |

**Хранение данных:** данные от агента хранятся только до следующей отправки (при новых данных старые стираются); история и даты формируются из отчётов в архиве. Подробно: [DATA-RETENTION-AGENT-VS-REPORTS.md](DATA-RETENTION-AGENT-VS-REPORTS.md).

Интеграция: агент отправляет пакеты в ingest (URL и контракт задаются главным порталом / конфигом); HIPAA SaaS хранит receipt’ы, отображает Inbox и связывает пакеты с тенантом и оценкой.
